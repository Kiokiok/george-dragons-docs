Battle

Noeud central orchestrant une bataille : spawn des pawns, gestion des états, requêtes de position et interactions sur le terrain. Classe partielle étendue par BattleInstanceMovement et BattleInstanceActions.

Méthodes publiques

  • void Init()
  • void BeginNewTurn()
  • bool GetPawnAtLocation(Vector2I, out BattlePawn)
  • bool TryUpdateObjectLocation(ITerrainObject, Vector3)
  • bool TrySwapObjects(ITerrainObject, ITerrainObject)
  • bool MovePawn(BattlePawn, List<Vector2I>)
  • bool TeleportPawn(BattlePawn, Vector2I, bool)
  • bool PushPawn(BattlePawn, Vector2I, int)
  • void DealDamage(DamagePacket)
  • void Heal(HealPacket)

Exemple d'usage

// Rechercher un pion à une position et infliger des dégâts
if (battle.GetPawnAtLocation(targetCell, out BattlePawn target))
{
    var packet = new DamagePacket(source, target, 25,
        DamageType.Magical);
    battle.DealDamage(packet);
}
// Déplacer un pion le long d'un chemin
var path = PathfinderOK.PathfindDumb<NavNode>(
    navGraph, startCell, destCell);
battle.MovePawn(pawn, path);

Explications du code

Pattern : Facade — BattleInstance est le point d'entrée unique pour toutes les opérations de combat (déplacement, dégâts, soins, téléportation). Les states et l'UI n'interagissent jamais directement avec les pawns ou le terrain — tout passe par cette façade, simplifiant le couplage.

Pattern : State Machine (IBattleState) — La bataille délègue sa logique à des états interchangeables (Positioning, PlayerTurn, EnemyTurn). Chaque état contrôle l'UI, les inputs et la progression — le passage d'un état à l'autre est géré par BattleInstance sans if/else massifs.

API Godot : partial class — Requis par le source generator Godot 4 C#. Permet aussi de découper la classe en BattleInstanceMovement et BattleInstanceActions pour organiser le code sans héritage.

Architecture — Créé par le noeud Battle (scène), consommé par tous les states, l'UI et l'IA. Possède les références au terrain (Subworld), au graphe de navigation et à la liste des pawns.

Battle

Noeud principal d'un combattant : gère les stats (HP/AP/MP), le mouvement, les capacités, les effets de statut et l'équipement.

Méthodes publiques

  • void Initialize(BattleInstance)
  • void InitializeFromCharacterData(BattleInstance, CharacterData)
  • bool CanBePlayed()
  • void OnBeginTurn()
  • void OnEndTurn(bool)
  • void RequestMoveAlongPath(List<Vector2I>)
  • bool CanMove()
  • int GetEffectiveMovementPoints()

Événements

  • event Action<DamagePacket> OnBeforeDealDamage
  • event Action<DamagePacket> OnBeforeTakeDamage
  • event Action<HealPacket> OnBeforeReceiveHeal
  • event Action<StatusEffect> OnBeforeReceiveEffect
  • event Action<MoveRequest> OnBeforeMove

Exemple d'usage

// Initialiser le pion avec les données personnage
pawn.InitializeFromCharacterData(battle, charData);
// Vérifier les PM effectifs (statuts + équipement)
int mp = pawn.GetEffectiveMovementPoints();
// Écouter les dégâts entrants via l'event pipeline
pawn.OnBeforeTakeDamage += (packet) => {
    if (packet.DamageType == DamageType.Physical)
        packet.FinalAmount -= 3; // armure
};
// Déplacer le pion sur la grille
pawn.RequestMoveAlongPath(path);

Explications du code

Pattern : Observer / Event Pipeline (16 events) — BattlePawn expose 16 événements Action<T> (OnBeforeDealDamage, OnBeforeTakeDamage, etc.) formant un pipeline d'interception. Les équipements et statuts s'abonnent pour modifier les paquets AVANT exécution — découplage total source/modificateur/cible.

Pattern : Composition — Au lieu d'hériter de sous-classes spécialisées, BattlePawn compose EquipmentController, StatusController et une liste d'AbilityRefs. Chaque composant gère son domaine indépendamment.

C# : Action<T> delegates — Utilisés au lieu des signaux Godot pour les événements internes de combat. Plus légers (pas de marshalling GDScript), typesafe, et permettent la modification in-place du paquet.

Architecture — Créé par BattleInstance via CharacterData. Consommé par les states (PlayerTurn, EnemyTurn), l'UI (PlayerTurnUI) et le système d'IA.

Battle

Classe de base abstraite pour toutes les capacités. Définit le contrat commun (portée, zone d'effet, déclenchement).

Méthodes

  • string GetName()
  • abstract string GetDescription()
  • abstract bool CanPawnCast(BattlePawn, BattleInstance)
  • abstract List<Vector2I> GetCastArea(...)
  • abstract List<Vector2I> GetEffectArea(...)
  • abstract void CastAbility(...)

Exemple — BouleDeFeu

public class BouleDeFeu : BaseAbility {
  public override List<Vector2I> GetCastArea(...) {
    var b = new ShapeBuilder(pawn.GetCellPos(world));
    b.AddSplat(4, 0, 0); // portée 4
    b.AddSingle(0, 0);
    b.CollapseSubstract(); // exclure soi-même
    return b.Collapse().cells;
  }
  public override List<Vector2I> GetEffectArea(...) {
    var b = new ShapeBuilder(target);
    b.AddSplat(1, 0, 0); // zone 1
    return b.Collapse().cells;
  }
}

Explications du code

Pattern : Template Method — BaseAbility définit le squelette d'une capacité (portée, zone d'effet, exécution) via des méthodes abstraites. Chaque aptitude concrète (BouleDeFeu, Soin, etc.) implémente uniquement les détails spécifiques sans réécrire la logique commune.

Pattern : Strategy — Chaque aptitude est interchangeable via la référence BaseAbility. Le système de combat ne connaît que l'interface abstraite — l'ajout d'une nouvelle aptitude ne nécessite aucune modification du code existant.

Architecture — Stockée dans CharacterData, référencée par BattlePawn.AbilityRefs. Exécutée par StatePlayerTurn (joueur) ou EnemyAISystem (IA). Utilise ShapeBuilder pour définir les zones.

Gère l'alternance des tours (Joueur ↔ Ennemi) avec pile d'overrides pour tours spéciaux.

Méthodes

  • BattleTurn GetCurrentTurn()
  • BattleTurn GetNextTurn()
  • void PushTurnOverride(BattleTurn)

Explications du code

Pattern : Stack-based Override — L'alternance standard (Joueur ↔ Ennemi) peut être court-circuitée via PushTurnOverride, qui empile un tour spécial. Permet des mécaniques comme « tour bonus » ou « contre-attaque » sans modifier la boucle principale.

Architecture — Consommé par BattleInstance pour déterminer quel state activer après chaque tour. Isolé de la logique de combat elle-même.

Data-classes représentant les intentions d'action. Chaque paquet contient source, cible, montant et peut être annulé avant exécution.

Propriétés (DamagePacket)

  • BattlePawn SourcePawn / TargetPawn
  • int BaseAmount / FinalAmount
  • DamageType DamageType
  • bool IsCancelled

Explications du code

Pattern : Data Transfer Object (DTO) mutable — Les paquets encapsulent une intention d'action (dégâts ou soin) avec source, cible et montant. La propriété IsCancelled et FinalAmount permettent aux événements du pipeline de modifier ou annuler l'action AVANT son exécution.

Pattern : Interceptor/Pipeline — Le paquet traverse les 16 events de BattlePawn (OnBeforeDealDamage → OnBeforeTakeDamage → application). Chaque abonné (statuts, équipements) peut modifier FinalAmount ou annuler, sans connaître les autres intercepteurs.

Méthodes (StatusController)

// Appliquer un poison empilable au pion cible
var poison = new PoisonEffect(duration: 3, dps: 5);
target.StatusController.AddEffect(poison, battle);
// Purger les débuffs via tag
int purged = pawn.StatusController

Pattern : Template Method + Strategy — StatusEffect est une classe abstraite dont chaque sous-classe (Poison, Stun, Buff) implémente ses propres règles d'empilement et de tick. Le StatusController itère sur tous les effets sans connaître leurs types concrets.

Pattern : Tag-based Filtering — RemoveEffectsByTag permet de purger des catégories entières d'effets (« Debuff », « DoT ») sans référencer les types concrets — extensible sans modifier le controller.

Architecture — StatusController est un composant de BattlePawn. Appelé par BattleInstance lors des débuts/fins de tour (ProcessTurnStart/End). Les aptitudes créent et appliquent les effets.

Battle

Système d'équipement par slots. Chaque équipement s'abonne aux événements du BattlePawn pour modifier les actions.

Méthodes

  • bool Equip(BaseEquipment)
  • bool Unequip(EquipmentType)
  • int GetMovementPointBonus()
  • int GetActionPointBonus()

Exemple — Épée avec bonus dégâts

public class SwordOfDamage : BaseEquipment {
  public override void OnEquip(BattlePawn pawn) {
    pawn.OnBeforeDealDamage += Boost;
  }
  void Boost(DamagePacket p) {
    p.FinalAmount += 5; // +5 dégâts
  }
}
// Équiper via le controller
pawn.EquipmentController.Equip(sword);

Explications du code

Pattern : Observer (abonnement aux events du pawn) — Chaque équipement s'abonne aux événements du BattlePawn (OnBeforeDealDamage, OnBeforeMove, etc.) lors de OnEquip et se désabonne lors de OnUnequip. Permet d'ajouter des comportements sans modifier le pawn.

Pattern : Open/Closed Principle — Nouveau type d'équipement = nouvelle sous-classe de BaseEquipment. Aucune modification du controller ou du pawn nécessaire — le système est ouvert à l'extension, fermé à la modification.

Architecture — EquipmentController est un composant de BattlePawn. Les données d'équipement viennent de CharacterData, configurées dans BarracksUI (caserne).

Pathfinding générique : BFS, A* et cellules atteignables sur NavGraphAdvanced.

Méthodes

  • static List<Vector2I> PathfindDumb<T>(...)
  • static bool IsPositionReachable<T>(...)
  • static List<Vector2I> GetAllReachableCells(...)

Explications du code

Pattern : Algorithmes génériques (BFS / A*) — PathfindDumb utilise BFS (sans heuristique) pour les chemins courts sur grille, tandis que GetAllReachableCells calcule toutes les cellules accessibles dans un budget de PM donné. La généricité <T> permet de fonctionner sur différents types de graphes.

C# : [MethodImpl(AggressiveInlining)] — Appliqué sur les méthodes chaudes du pathfinding pour réduire l'overhead des appels de fonction lors du parcours de milliers de noeuds.

Architecture — Utilisé par BattleInstance (MovePawn), StatePlayerTurn (prévisualisation de mouvement) et EnemyAISystem (planification). Opère sur le NavGraphAdvanced du Subworld.

Anime un Node3D le long d'un chemin de cellules en interpolant sa position.

Méthodes

  • bool Update(float _dt)
  • void BeginFollowPath(List<Vector2I>)
  • void StopOngoingMovement()

Explications du code

Pattern : Coroutine-like Update — Update(dt) retourne un booléen indiquant si le mouvement est terminé, permettant au state de la bataille de poll chaque frame sans callback ni coroutine. Approche idéale pour les animations frame-by-frame dans Godot.

API Godot : interpolation de position — Anime le Node3D le long du chemin de cellules en interpolant GlobalPosition entre chaque point, avec conversion cellule → position monde via VectorHelpers.

Battle

IA ennemie utilisant un algorithme minimax simplifié pour calculer le meilleur plan d'action.

Méthodes

  • TurnPlan CalculateBestTurn(BattlePawn)
  • void ExecuteTurnPlan(BattlePawn, TurnPlan)

Explications du code

Pattern : Minimax simplifié — L'IA évalue les actions possibles (déplacement + aptitude) via une fonction heuristique (proximité cible, dégâts potentiels, survie) et choisit le plan avec le meilleur score. Simplifié car la profondeur est limitée à 1 tour.

Pattern : Memento (TurnPlan) — Le résultat du calcul est encapsulé dans un TurnPlan immutable, séparant la phase de décision (CalculateBestTurn) de la phase d'exécution (ExecuteTurnPlan).

Architecture — Appelé par StateEnemyTurn pour chaque pion ennemi. Consomme PathfinderOK et BaseAbility pour évaluer les options.

IBattleState : StatePlayerPositioning, StatePlayerTurn et StateEnemyTurn.

Interface IBattleState

  • void OnBeginState()
  • void OnProcessState(float _dt)
  • void OnExitState()
  • bool IsStateFinished(float _dt)

Explications du code

Pattern : State (IBattleState) — Chaque phase du combat implémente la même interface. BattleInstance délègue Begin/Process/Exit sans savoir quel état est actif — ajout d'une nouvelle phase = nouvelle classe, zéro modification du code existant.

Pattern : Sub-State Machine (StatePlayerTurn) — Le tour du joueur contient lui-même une pile de sous-états (sélection pion, sélection aptitude, ciblage, confirmation) gérée via une Stack interne, permettant l'annulation en « dépilant ».

Architecture — Les states sont créés par BattleInstance. PlayerTurn contrôle PlayerTurnUI, EnemyTurn délègue à EnemyAISystem, Positioning gère le placement initial.

Singleton central ; gère les transitions d'états (MainMenu, Camp, Battle), instancie/détruit les scènes.

Méthodes

  • static PackedScene[] GetPlayerPawnsPrefabs()
  • static PackedScene GetPlayerPawnTemplate()

Événements

  • static Action<GameState> onGameStateChanged

Exemple d'usage

// Changer d'état de jeu (détruit la scène
// actuelle et instancie la nouvelle)
Game.RequestChangeGameState(
    GameState.Battle);
// Écouter les transitions d'état
Game.onGameStateChanged += (state) => {
    if (state == GameState.Camp)
        TeamManager.Initialize();
};

Explications du code

Pattern : Singleton (static) — Game est entièrement statique — pas d'instance, accès global via Game.RequestChangeGameState(). Évite l'injection de dépendances dans un jeu où un seul état global existe à la fois.

Pattern : State Machine + Observer — L'événement Action<GameState> onGameStateChanged notifie tous les abonnés (UI, musique, systèmes) lors d'un changement d'état. Le découplage permet d'ajouter des réactions sans modifier Game.

API Godot : PackedScene + instantiation — Charge et instancie les scènes (Camp, Battle, WorldMap) dynamiquement via PackedScene.Instantiate(), détruisant la scène précédente avec QueueFree().

Architecture — Point d'entrée du jeu. Tous les systèmes (Battle, Camp, UI) dépendent de Game pour les transitions. Possède les références aux PackedScene des personnages.

États de jeu : None, MainMenu, WorldMap, Camp, Battle, Exploration.

Explications du code

C# : enum simple — Définit les états possibles du jeu. Utilisé par Game pour le switch de scènes et par l'UI (EscapeMenu) pour déterminer les options de navigation disponibles.

Noeud principal du mode bataille ; gère génération terrain, création BattleInstance et curseur.

Méthodes
